1. PROJECT OVERVIEW & RESEARCH THESIS
================================================================================

CHOSEN DIRECTION (from conversation with Tomer):
  "Towards Super Interpretability" — LLM-based learning algorithms that can
  explain themselves.

THESIS:
  LLM-based learning algorithms (like LLM-PV) enjoy THREE unique aspects of
  interpretability that traditional deep learning completely lacks:

  (1) CODE INTERPRETABILITY:
      The LLM generates explicit Python code (if/else, thresholds, formulas)
      rather than opaque weight matrices. The output IS the explanation.

  (2) LEARNING TRACE INTERPRETABILITY:
      The iterative history of how the LLM arrived at the final code — what
      it tried, what failed, what it changed — is a readable decision log.

  (3) THESIS-BASED SELF-EXPLANATION (revised per Tomer meetings 2/15, 2/24):
      After classification, the LLM produces a structured THESIS for each
      prediction: a short set of CONDITIONS that lead to a LABEL. This thesis
      defines a subset of the data, and we can QUANTITATIVELY measure:
      (a) how large/general that subset is (coverage), and
      (b) how well the thesis reflects real structure in the training data
          (faithfulness — measured against GROUND TRUTH labels), and
      (c) optionally, how well the thesis aligns with Code0's specific
          behavior (faithfulness-code0 — secondary metric).
      The LLM is both the learner AND the explainer — same algorithm that
      wrote the code can explain it. This is unprecedented.

  The key constraint: explanations must be DATA-GROUNDED and CODE-ALIGNED.
  The thesis must:
  - Be SHORT and human-interpretable (low entropy)
  - Be MEANINGFUL (not just restating the code)
  - Be FAITHFUL to the DATA (the subset it defines should have consistent
    ground truth labels — this is the PRIMARY faithfulness metric)
  - Be FAITHFUL to the CODE (the subset should align with Code0's
    behavior — this is the SECONDARY faithfulness metric)
  - Define conditions that hold for the specific sample AND generalize
    to a significant portion of the training data
  - NOT rely on comments — the code itself must have inherent structure
    that supports explanation

WHY THIS MATTERS:
  - Explainable AI (XAI) is one of the biggest unsolved problems in ML
  - Existing XAI methods (SHAP, LIME, attention maps) are post-hoc
    approximations of opaque models
  - LLM-based learners are the FIRST learning algorithms where the model
    can genuinely explain its own reasoning from the data
  - Potentially monetizable (Tomer's words) — industry cares deeply about
    interpretability for compliance, trust, debugging
  - Novel contribution: no one has tried to have a conversation with a
    learning algorithm about what it learned, because until LLMs, this was
    impossible

STARTING POINT:
  Tabular data classification (the codebase already has 5 tabular datasets:
  adult_income, mushroom, cdc_diabetes, htru2, chess). Tabular is ideal
  because:
  - Features have semantic meaning (age, income, etc.)
  - Decision boundaries are verifiable against the data
  - If/else code is naturally interpretable for tabular tasks
  - Easier to verify LLM's explanations against actual data statistics

ALTERNATIVE DIRECTION (NOT chosen, but documented for reference):
  "Distribution-Aware Algorithm Design with LLMs" — designing algorithms
  tailored to specific problem distributions (SAT subsets, TSP variants).
  This is the clearer, easier-to-evaluate project but less novel. Tomer
  said both are great; Lucas chose interpretability.

NEW COMPONENTS (in order of priority):

  1. INTERPRETABILITY-AWARE PROMPTING MODULE [PHASE 1 — COMPLETE]
     - Modified prompts that instruct the LLM to generate "explainable" code
     - Raw generated code may include comments, but Code0 is normalized by
       stripping comments/docstrings before compile/eval so the evaluated
       code remains self-explanatory
     - Prompt tells LLM: "Your code should have inherent structure so that
       if asked a question, you can answer just by looking at the code"
     - Comparison: standard prompt vs. interpretability-aware prompt
     - KEY FINDING FROM PHASE 1: interpretability prompts ~doubled if-statements
       and code length; code became more self-explanatory

  2. THESIS-BASED SELF-EXPLANATION SYSTEM [PHASE 2 — COMPLETE]
     - Given Code0 (generated classifier) and a test sample x with predicted
       label, the LLM produces a THESIS of the form:
         { conditions → label }
       Example: "If x1 > 5 and x3 < 2, then label = 1"
     - The thesis defines a set A = {x | cond(x) = true}
     - A_S = A ∩ S (training samples satisfying the conditions)
     - The LLM also generates VERIFICATION CODE (Code1) that checks:
       * Whether a given sample satisfies the thesis conditions
       * Can be run on all training data to measure A_S
     - The thesis must be SHORT, MEANINGFUL, and FAITHFUL to the code

  3. DUAL-METRIC QUANTITATIVE EVALUATION (revised per Tomer meeting 2/24)
     - METRIC 1 — COVERAGE: E_{x~test}[ I(x ∈ A^x) · (|A^x_S| / |S|) ]
       How large is the subset described by the thesis? Bigger = better.
       Thesis must not be hyperspecific to just one sample.
       Baseline: ~0.5 (half the data is class 1)
       Best case: captures ALL same-class samples
       STATUS: Correctly implemented in Phase 2 as coverage_eq. ✓

     - METRIC 2a — FAITHFULNESS (GROUND TRUTH) [PRIMARY]:
       E[ Pr[y_true(x_i) = ŷ_0(x) | x_i ∈ A_S] ]
       Among training samples in A_S, what fraction have the same GROUND
       TRUTH LABEL as Code0's prediction on x? Measures whether the thesis
       identifies a subset of the data where the real labels support the
       prediction. This is the metric that makes claims about the DATA.
       Baseline: 1.0 (thesis claims perfect faithfulness)
       We measure how close to 1.0 we actually get.
       STATUS: NOT YET IMPLEMENTED. Phase 2 used Code0 labels instead
       of ground truth. To be corrected in Phase 2.5.

     - METRIC 2b — FAITHFULNESS (CODE0) [SECONDARY]:
       E[ Pr[ŷ_0(x_i) = ŷ_0(x) | x_i ∈ A_S] ]
       Among training samples in A_S, what fraction are classified THE
       SAME WAY as x by Code0? Measures alignment between thesis and
       Code0's specific behavior. Useful secondary metric — tells you
       how well the thesis reads Code0 — but not the primary metric
       because it doesn't make claims about the underlying data.
       STATUS: This is what Phase 2 implemented as "faithfulness".
       To be RENAMED to faithfulness_code0 in Phase 2.5.

     These metrics have CLEAR BASELINES and can be used to:
     - Compare different LLM models against each other
     - Optimize prompting strategies
     - Benchmark interpretability quantitatively
     - The GAP between faithfulness_gt and faithfulness_code0 reveals
       how much of Code0's behavior is supported by real data structure

  4. EXTERNAL VERIFIER (optional, for code correctness)
     - A frontier LLM (e.g., GPT-5) verifies that Code1 actually checks
       the conditions stated in the thesis
     - OR: automated system that parses conditions and generates verification
       code (since conditions → code is straightforward)
     - Tomer: "We can trust the verifier for now — if someone provides
       something better, we'll use that"

  5. LEARNING TRACE LOGGER (future phase)
     - Enhanced logging that captures iterative refinement process
     - What the LLM changed between iterations and why

  6. MODEL COMPARISON FRAMEWORK
     - Run the same thesis-based evaluation across different LLMs
     - Each LLM is a "learner" baseline; verifier is always the same
     - Compare coverage and faithfulness across models

  7. BATCHED/ITERATIVE CODE0 TRAINING [PHASE 3 — NOT STARTED]
     Currently, Code0 generation passes ALL training samples (e.g., 200) to
     the LLM in a single prompt. Tomer proposed an iterative approach:

     - Give the LLM a BATCH of training samples (e.g., 16-32)
     - LLM generates initial Code0 from the first batch
     - Next iteration: pass the NEXT batch + the CURRENT code
     - LLM updates the code to also handle the new batch
     - Repeat for multiple epochs (iterate over all batches multiple times)

     This is analogous to SGD: instead of full-batch gradient descent (all 200
     samples at once), we do mini-batch updates. Benefits:
     (a) Scales beyond context window limits (200 samples = ~25K tokens input;
         with batching, each call uses only ~2K-5K tokens for the batch)
     (b) May produce BETTER Code0: easier for LLM to fit 16 samples well than
         200 at once. 10 batches of 16 may outperform 200 at once.
     (c) Context-window agnostic: can compare models with different context
         lengths (Claude 1M vs GPT 400K) fairly
     (d) Enables epochs: iterate over batches multiple times like neural net
         training, potentially reaching 100% train accuracy

     Implementation in runner.py:
     - New mode: --batch-size 16 --epochs 3
     - Each "iteration" = one LLM call with (current_code + new_batch)
     - Prompt: "Here is the current classifier code and a new batch of
       training examples. Update the code to ALSO correctly classify these
       new samples while maintaining accuracy on previous samples."
     - Track per-iteration metrics: batch_acc, cumulative_train_acc

     From Tomer: "Keep doing that until you fit the data."

  8. NEW METRICS [PHASE 3 — NOT STARTED]
     Per Tomer's direction, add:
     (a) Train accuracy: Code0 accuracy on training set (currently only
         val_acc and test_acc are tracked). MUST be on record.
     (b) Code0 token count: absolute length of Code0 in tokens
     (c) Code1 token count: absolute length of Code1 in tokens
     NOTE: Track absolute counts, NOT ratios. Ratios can be gamed by
     inflating Code0 with irrelevant code. (Tomer: "The ratio doesn't
     tell you anything. Just the total [tokens].")

  9. CODE TRACE VISUALIZATION [PHASE 5 — NOT STARTED]
     For each thesis, trace which lines of Code0 are executed by samples
     in the cover set (A_S). Color each line by visit frequency:
     - Darker color = more samples traverse this line
     - Shows "line importance" for the thesis
     - Validates that thesis conditions align with actual code execution
     Per Tomer: "We color each line, give it a higher shade if more
     samples from the cover went there."

PHASE 2: THESIS-BASED SELF-EXPLANATION SYSTEM (Days 9-18) [COMPLETE]
----------------------------------------------------------------------
Goal: Build the core thesis-based explanation framework. Given Code0
      (the LLM's generated classifier) and a test sample x, the LLM
      must produce a structured thesis explaining WHY x was classified
      the way it was. The thesis is quantitatively evaluated using
      metrics derived from Tomer's whiteboard formalization.

  *** MATHEMATICAL FRAMEWORK (from Tomer's whiteboard, 2/15/2026) ***
  *** (Faithfulness corrected per Tomer meeting 2/24/2026)        ***

  DEFINITIONS:
    - S = training set
    - Code0 = classification code generated by LLM from training data
    - x = a test sample passed through Code0, producing a label
    - Thesis = structured explanation: {conditions → label}
    - A = {x | cond(x) = true}  — set of ALL samples satisfying conditions
    - A_S = A ∩ S  — training samples satisfying conditions
    - A^x = the set A proposed specifically for test sample x
    - ŷ_0(·) = Code0's prediction function
    - y_true(·) = ground truth label from the dataset

  METRIC 1 — COVERAGE:
    E_{x~test}[ I(x ∈ A^x) · (|A^x_S| / |S|) ]

    Interpretation: For each test sample x, IF x itself satisfies the
    proposed conditions (indicator I), THEN measure what fraction of the
    training set also satisfies those conditions. Average over test samples.
    - If x doesn't satisfy its own thesis conditions → indicator is 0,
      the thesis is completely wrong, contributes nothing.
    - If conditions are too specific (only 1 training sample matches) →
      |A^x_S|/|S| is tiny → low coverage → bad thesis.
    - If conditions are broad and general → high coverage → good thesis.
    - BASELINE: ~0.5 (trivially, "label = 1" covers half the data)
    - BEST CASE: thesis captures ALL same-class samples

  METRIC 2a — FAITHFULNESS, GROUND TRUTH [PRIMARY]:
    E[ Pr[y_true(x_i) = ŷ_0(x) | x_i ∈ A_S] ]

    Interpretation: Among all training samples that satisfy the thesis
    conditions (A_S), what fraction have a GROUND TRUTH LABEL that
    matches Code0's prediction on the test sample x?
    - This measures whether the thesis identifies real structure in the
      data — a subset where the actual labels consistently support the
      prediction.
    - If thesis says "these conditions → label 1" but the real labels
      in A_S are mixed → faithfulness_gt is low → bad thesis.
    - If ALL samples in A_S genuinely have label 1 → 1.0 → perfect.
    - BASELINE: 1.0 (the thesis CLAIMS 100% faithfulness)
    - We measure how close to 1.0 we actually achieve.

    WHY GROUND TRUTH (from Tomer 2/24):
    "We want to say: this sample is labeled L because there is a real
    group in the training data that shares these properties and they
    are ALL genuinely labeled L. We want to make claims about the DATA,
    not explain what Code0 is doing internally."

  METRIC 2b — FAITHFULNESS, CODE0 [SECONDARY]:
    E[ Pr[ŷ_0(x_i) = ŷ_0(x) | x_i ∈ A_S] ]

    Interpretation: Among all training samples that satisfy the thesis
    conditions (A_S), what fraction are classified THE SAME WAY as x
    by Code0? This measures alignment between thesis and code behavior.
    - Useful as a secondary metric: tells you how well the thesis
      captures Code0's decision logic specifically.
    - Tomer: "not a terrible metric" — keep it, but not as primary.
    - This was what Phase 2 originally computed as "faithfulness".

  THE FAITHFULNESS GAP:
    faithfulness_code0 - faithfulness_gt
    - Small gap → Code0 is well-aligned with ground truth on this subset
    - Large gap → Code0 has learned patterns not supported by real labels
    - This diagnostic metric reveals noise in Code0's learning

  WHY THESE METRICS ARE POWERFUL (Tomer's insight):
    - Both have CLEAR BASELINES → we know what "good" looks like
    - Both are FULLY AUTOMATED → no human judgment needed
    - They capture a TRADEOFF: broad conditions (high coverage) may
      sacrifice faithfulness; narrow conditions (high faithfulness)
      sacrifice coverage
    - They enable QUANTITATIVE COMPARISON between LLM models
    - They define "interpretability" as a measurable quantity for the
      first time in this context
    - The dual faithfulness (GT vs Code0) reveals how much the code's
      behavior is justified by the actual data

  *** END MATHEMATICAL FRAMEWORK ***

  Step 2.1: Implement Code0 Normalization (Manual Strip Before Eval)
    Policy update:
    - Do NOT require "no comments" in the generation prompt.
    - Keep interpretability prompting focused on structure and meaningful
      conditions.
    - After code extraction, normalize Code0 by stripping comments,
      docstrings, and other non-executable explanation text before
      compile/eval.
    - Use normalized Code0 for all metrics/logged evaluation artifacts.

    Why:
    - Prevents avoidable compile failures from style violations.
    - Preserves apples-to-apples performance with prior benchmarks.
    - Keeps the evaluated model comment-free without needing hard prompt
      constraints.

  Step 2.2: Design the Thesis Prompt
    After Code0 is generated and a sample x is classified, prompt the LLM:

    THESIS GENERATION PROMPT:
    """
    You previously generated the following classification code from
    training data:

    [CODE0]

    I ran the following sample through your code:
    Sample: [x0=val, x1=val, x2=val, ...]
    Your code classified it as: [LABEL]

    Explain WHY this classification was made. Your explanation must be
    a structured thesis of the following form:

    THESIS:
    Conditions: [list of conditions on the features, e.g., "x1 > 5 AND x3 < 2"]
    Label: [the predicted label, e.g., 1]

    Requirements:
    - The conditions must be SHORT and human-interpretable
    - The conditions must hold for the given sample
    - The conditions must reflect what your CODE actually does (not just
      random true facts about the data)
    - The conditions should define a MEANINGFUL subset of the data — not
      just this one sample, but a general pattern
    - Every sample that satisfies these conditions should be classified
      as [LABEL] by your code (or as close to 100% as possible)
    - The subset should be as LARGE as possible while maintaining accuracy

    Output your thesis in this exact JSON format:
    {
      "conditions": "x1 > 5 AND x3 < 2",
      "label": 1
    }
    """

  Step 2.3: Build Verification Code Generation (Code1)
    The LLM generates Code1 — a simple function that checks whether a
    given sample satisfies the thesis conditions.

    VERIFICATION CODE PROMPT:
    """
    You proposed the following thesis:
    Conditions: [CONDITIONS]
    Label: [LABEL]

    Write a Python function `check_conditions(x)` that takes a sample x
    (a list of feature values) and returns True if the sample satisfies
    ALL of the conditions in the thesis, and False otherwise.

    Output ONLY the function (any comments/docstrings will be stripped
    before execution).
    """

    ALTERNATIVE APPROACH (simpler, Tomer suggested this is fine):
    - Parse the conditions string automatically
    - Generate the verification code ourselves (no LLM needed for this)
    - Since conditions are simple boolean expressions (x1 > 5 AND x3 < 2),
      we can eval() them or parse them into a function
    - This removes one source of error (LLM-generated verifier bugs)

  Step 2.4: Implement the Evaluation Pipeline
    Create: program_synthesis/thesis_evaluator.py

    Class: ThesisEvaluator
      __init__(self, code0_fn, train_data, train_labels, test_data)

      Methods:
        - evaluate_thesis(sample_x, thesis_conditions, thesis_label):
            1. Check if x satisfies conditions → indicator I(x ∈ A^x)
            2. Find A_S: all training samples satisfying conditions
            3. Compute coverage: |A_S| / |S|
            4. Compute faithfulness_gt: fraction of A_S where ground truth
               label matches ŷ_0(x) [PRIMARY]
            5. Compute faithfulness_code0: fraction of A_S where Code0
               gives same label as ŷ_0(x) [SECONDARY]
            6. Return: {
                 sample: x,
                 predicted_label: ŷ_0(x),
                 thesis_conditions: str,
                 thesis_label: int,
                 x_satisfies_conditions: bool,
                 coverage: float,
                 A_S_size: int,
                 S_size: int,
                 faithfulness_gt: float,       # PRIMARY
                 agreement_count_gt: int,
                 faithfulness_code0: float,     # SECONDARY
                 agreement_count_code0: int,
               }

        - evaluate_over_test_set(num_samples=None):
            For each test sample (or a subset):
            1. Run Code0 on sample → get label
            2. Ask LLM for thesis
            3. Parse thesis (conditions + label)
            4. Generate or parse verification code
            5. Evaluate thesis
            6. Aggregate:
               - Mean coverage (Metric 1)
               - Mean faithfulness_gt (Metric 2a — PRIMARY)
               - Mean faithfulness_code0 (Metric 2b — SECONDARY)
               - % of samples where x ∈ A^x (thesis applies to own sample)
               - Distribution of A_S sizes
               - Faithfulness gap (code0 - gt)
            7. Return: ThesisEvaluationReport

        - parse_conditions(conditions_str) -> Callable:
            Parse a condition string like "x1 > 5 AND x3 < 2" into a
            callable function that takes a sample and returns bool.
            Handle: >, <, >=, <=, ==, AND, OR operators.

    Class: ThesisEvaluationReport
      Fields: per_sample_results, mean_coverage, mean_faithfulness_gt,
              mean_faithfulness_code0, condition_validity_rate, summary_stats
      Methods:
        - to_json()
        - to_csv()
        - to_markdown()

  Step 2.5: Implement the Full Pipeline Orchestrator

PHASE 2.5: EQUATION CORRECTIONS & DUAL FAITHFULNESS (Days 19-20)
----------------------------------------------------------------------
Goal: Correct the faithfulness metric from Code0-based to ground-truth-
      based (per Tomer meeting 2/24), rename old metric as secondary,
      and update experimental parameters for cost efficiency.

  Context: Phase 2 implementation computed faithfulness by comparing
  Code0's predictions on training samples to Code0's prediction on the
  test sample. Tomer identified this as incorrect — the PRIMARY
  faithfulness must compare against ground truth training labels to
  make claims about the data's real structure, not just Code0's
  internal consistency.

  WHAT'S CORRECT (keep as-is):
    - coverage_eq: 1[c_i(x_i)=1] · |A_{x_i}| / |S|  ✓
    - coverage_ratio: |A_{x_i}| / |S| (secondary)  ✓
    - x_in_A_rate  ✓
    - A_{x_i} computation  ✓
    - All aggregation logic (structurally)  ✓

  WHAT'S WRONG (must change):
    - agreement_count uses ŷ_0(x) = ŷ_0(x_i) [Code0 vs Code0]
      SHOULD BE: y_true(x) = ŷ_0(x_i) [ground truth vs Code0]
    - faithfulness inherits this error

  Step 2.5.1: Add Ground-Truth Faithfulness (PRIMARY METRIC)
  ---------------------------------------------------------------
  File: program_synthesis/thesis_evaluator.py

  Add new per-case metric:

    agreement_count_gt_i = Σ_{x ∈ A_{x_i}} 1[y_true(x) = ŷ_0(x_i)]

    Where:
      y_true(x) = the GROUND TRUTH label of training sample x
      ŷ_0(x_i)  = Code0's prediction on the test sample x_i

    faithfulness_gt_i =
      agreement_count_gt_i / |A_{x_i}|   if |A_{x_i}| > 0
      None                                 if |A_{x_i}| = 0

  Implementation: one new counter in the same loop that already
  iterates over A_{x_i}. The training data already includes ground
  truth labels (y_train). Minimal code change.

  Step 2.5.2: Rename Existing Faithfulness (SECONDARY METRIC)
  ---------------------------------------------------------------
  File: program_synthesis/thesis_evaluator.py

  Rename current metrics to clarify they measure Code0 alignment:

    OLD NAME                              → NEW NAME
    agreement_count_i                     → agreement_count_code0_i
    faithfulness_i                        → faithfulness_code0_i
    mean_faithfulness_defined_only        → mean_faithfulness_code0_defined
    mean_faithfulness_all_missing_as_zero → mean_faithfulness_code0_all_zero

  Tomer: "I'm okay with calculating that too." It's a valid secondary
  metric — it tells you how well the thesis captures Code0's behavior.

  Step 2.5.3: Add Ground-Truth Aggregates

      None                                            if |I_gt| = 0

--- CODE0 NORMALIZATION POLICY (Phase 2 — Manual Strip) ---

Updated approach for thesis-compatible code generation/evaluation:

```
# Prompt remains interpretability-focused; no hard "no-comments" constraint.
# After extraction, always sanitize generated code before compile/eval.

def sanitize_generated_code(code: str) -> str:
    code = normalize_generated_code(code)       # unwrap JSON/fences/indent
    code = strip_comment_tokens(code)           # remove # comments
    code = strip_docstrings_via_ast(code)       # remove module/function/class docstrings

```

Policy:
  - The model may emit comments/docstrings in raw output.
  - Code0 used for evaluation is always sanitized first.
  - Interpretability quality is judged on executable structure and metrics,
    not on raw comment presence.

--- THESIS GENERATION (Phase 2) ---

Key design decisions (from Tomer 2/15):
  - The SAME LLM that generated Code0 produces the thesis
    This is critical — "it's both learning and making the code, and then
    being able to explain it — all the same algorithm" (Tomer)
  - The LLM has access to the training data (it's in context)
  - Post-hoc justification is OK — the LLM looks at the data after
    writing code and explains. "As long as it is measured against the
    code" (Tomer)
  - Thesis must be SHORT: "you can do so much with short text — the
    entropy is low" (Tomer)
  - No data percentages in initial thesis — just: conditions → label
    "Let's start with 'all of the time' — every sample satisfying
    conditions gets this label" (Tomer). Can extend to percentages later.

Thesis format (strict JSON output):
  {
    "conditions": "x1 > 5 AND x3 < 2",
    "label": 1
  }

The conditions define:
  A = {x | cond(x) = true}  — the set of all points matching conditions
  A_S = A ∩ S  — training points matching conditions

--- CONDITION PARSING (Phase 2) ---

Since thesis conditions are simple boolean expressions, we can parse
them automatically without needing an LLM verifier:

Supported operators: >, <, >=, <=, ==, !=
Supported connectives: AND, OR
Supported operands: x0, x1, ..., xN (feature indices), numeric constants

Parser output: a Python callable check_conditions(x) -> bool

Example:
  Input: "x1 > 5 AND x3 < 2"
  Output: lambda x: x[1] > 5 and x[3] < 2

This is much more reliable than having the LLM generate verification code.
If conditions are too complex to parse automatically, fall back to
LLM-generated Code1 (with optional external verifier).

--- METRIC COMPUTATION (Phase 2-3, corrected Phase 2.5) ---

METRIC 1 — COVERAGE:
  For each test sample x:
    1. Generate thesis → get conditions
    2. Parse conditions into check_conditions()
    3. Check: does x satisfy conditions? → I(x ∈ A^x)
    4. Count training samples satisfying conditions: |A^x_S|
    5. Contribution = I(x ∈ A^x) × (|A^x_S| / |S|)
  Average over all test samples → COVERAGE score

  Interpretation:
    - 0.0 = no thesis ever applies to its own sample
    - 0.5 = thesis covers half the training data (trivial baseline)
    - Higher = thesis describes a large, general pattern
    - Note: coverage above 0.5 means the thesis is more informative
      than the trivial "label = majority class" thesis

METRIC 2a — FAITHFULNESS, GROUND TRUTH [PRIMARY]:
  For each test sample x (where I(x ∈ A^x) = 1):
    1. Find A_S (training samples satisfying conditions)
    2. For each sample in A_S, check its GROUND TRUTH label
    3. Count how many have y_true(x_i) == ŷ_0(x) (ground truth matches
       Code0's prediction on the test sample)
    4. Faithfulness_gt = agreement_count_gt / |A_S|
  Average over applicable test samples → FAITHFULNESS_GT score

  Interpretation:
    - 1.0 = thesis identifies a subset where all real labels match
    - 0.5 = thesis subset has mixed real labels → bad explanation
    - This makes claims about the DATA: "this prediction is correct
      because the data genuinely has this structure"

METRIC 2b — FAITHFULNESS, CODE0 [SECONDARY]:
  For each test sample x (where I(x ∈ A^x) = 1):
    1. Find A_S (training samples satisfying conditions)
    2. Run Code0 on each sample in A_S
    3. Count how many get ŷ_0(x_i) == ŷ_0(x) (Code0 agrees with itself)
    4. Faithfulness_code0 = agreement_count_code0 / |A_S|
  Average over applicable test samples → FAITHFULNESS_CODE0 score

  Interpretation:
    - 1.0 = thesis perfectly describes Code0's decision logic
    - Useful for understanding Code0's internal consistency
    - Tomer: "not a terrible metric" — keep as secondary

THE FAITHFULNESS GAP:
  mean_faithfulness_code0 - mean_faithfulness_gt
  - Reveals how much of Code0's behavior is NOT supported by real data
  - Small gap → Code0 aligns well with ground truth on thesis subsets
  - Large gap → Code0 has learned patterns beyond what the data supports

6. EVALUATION & BENCHMARKING STRATEGY
   (Revised per Tomer meetings 2/15 and 2/24)
================================================================================

PRIMARY EVALUATION:
  Goal: Show that LLM-based learning algorithms provide QUANTITATIVELY
  MEASURABLE self-explanations, using Tomer's thesis-based framework.

  Three Core Metrics (two primary, one secondary):
  1. COVERAGE — how general/broad are the thesis conditions?  [PRIMARY]
     E_{x~test}[ I(x ∈ A^x) · (|A^x_S| / |S|) ]
     Baseline: ~0.5 (trivially covering half the data)
     Goal: significantly above baseline

  2a. FAITHFULNESS (GT) — does the thesis reflect real data structure?  [PRIMARY]
      E[ Pr[y_true(x_i) = ŷ_0(x) | x_i ∈ A_S] ]
      Baseline: 1.0 (thesis claims perfect faithfulness)
      Goal: as close to 1.0 as possible

  2b. FAITHFULNESS (CODE0) — does the thesis align with Code0?  [SECONDARY]
      E[ Pr[ŷ_0(x_i) = ŷ_0(x) | x_i ∈ A_S] ]
      Useful for understanding Code0's internal consistency
      The gap between 2b and 2a reveals Code0's noise

  Supporting Metrics:
  3. Classification accuracy (must be competitive — within 5% of baselines)
  4. Validity rate — % of samples where x ∈ A^x (thesis applies to itself)
  5. Mean |A_S| — typical size of the thesis-defined subset
  6. Faithfulness gap — mean(faithfulness_code0 - faithfulness_gt)

EXPERIMENTAL PARAMETERS (revised per Tomer 2/24):
  Development runs (Phase 2.5 through Phase 3):
    - 1 seed
    - 10 Code0 attempts (single pass, keep best)
    - 25+ test samples per dataset
    - Rationale: minimize cost while building pipeline

  Final paper runs (Phase 4 and Phase 6):
    - 5-10 seeds
    - 10 Code0 attempts per seed
    - Full test set (50-100 samples)
    - Rationale: statistical rigor for publication

DATASETS FOR EVALUATION:
  Use the 5 tabular datasets already in the codebase:
  - adult_income (fn_m): 14 features, well-studied, good baseline comparisons
  - mushroom (fn_n): 20 features, high baseline accuracy, interesting rules
  - cdc_diabetes (fn_o): 21 features, noisy real-world health data
  - htru2 (fn_p): 8 features, clean scientific data
  - chess (fn_q): 35 features, high-dimensional

BASELINES:
  1. Different LLM models (GPT-5, GPT-5-mini, GPT-4o, etc.) as learners
     - Same verifier for all → isolates learner quality
  2. Decision Tree thesis extraction (tree path = thesis)
  3. LLM-PV standard (no interpretability prompting) — can it still explain?
  4. Trivial baselines:
     - "Always label X" (coverage = 0.5, faithfulness_gt = ~0.5)
     - "Exact sample match" (coverage = 1/|S|, faithfulness_gt = 1.0)
     - "Code0 as thesis" (coverage = best, faithfulness_gt = Code0 accuracy)
     - Random conditions (low faithfulness)

KEY HYPOTHESES TO TEST:
  H1: LLM-PV with interpretability prompting achieves coverage > 0.3
      AND faithfulness_gt > 0.75 (meaningfully above trivial baselines)
  H2: Interpretability prompting does NOT significantly hurt accuracy
      (within 3% of standard LLM-PV) — ALREADY CONFIRMED IN PHASE 1
  H3: The LLM produces broader theses (higher coverage) than decision
      tree path-based explanations
  H4: Faithfulness_gt improves when Code0 is more accurate (subsidiary
      relationship between performance and explainability)
  H5: Iterative refinement improves both accuracy AND thesis quality,
      and the faithfulness gap shrinks over iterations
  H6: The coverage-faithfulness tradeoff reveals meaningful differences
      between LLM models as learners
  H7: The faithfulness gap (code0 - gt) varies across datasets, revealing
      which domains have cleaner learnable structure

7. PAPER OUTLINE & NARRATIVE
   (Revised per Tomer meetings 2/15 and 2/24)
================================================================================

Title: "Towards Super Interpretability: Self-Explaining Learning Algorithms
        via Large Language Models"

Abstract:
  We introduce the concept of "super interpretability" in machine learning:
  learning algorithms that can explain their own reasoning through
  structured, verifiable self-explanations. Unlike traditional XAI methods
  (SHAP, LIME) that provide post-hoc approximations of opaque models,
  LLM-based learners generate explicit, human-readable classification code
  AND can produce structured theses explaining each prediction. We propose
  a thesis-based evaluation framework with quantitative metrics —
  COVERAGE (how general is the explanation?) and FAITHFULNESS (how well
  does the explanation align with the data's real structure?) — both with
  clear baselines. We demonstrate that LLM-based classifiers achieve
  competitive accuracy on tabular benchmarks while providing self-
  explanations that significantly exceed trivial baselines on both metrics.
  We further show that the gap between data-grounded faithfulness and
  code-aligned faithfulness reveals meaningful properties of the learning
  algorithm's behavior.

1. Introduction
   - The interpretability crisis in ML
   - Why existing XAI is insufficient (post-hoc, approximate, no dialogue)
   - The unique opportunity: LLMs as BOTH learner AND explainer
   - Key insight: same algorithm writes code and explains it
   - Our thesis-based framework for measurable interpretability

2. Related Work
   - Explainable AI: SHAP, LIME, attention, concept-based explanations
   - Program synthesis: neural-guided, LLM-based
   - LLM-PV and related code generation for ML
   - Interpretable ML: decision trees, rule lists, GAMs
   - Measuring interpretability: existing attempts and their limitations

3. Method: Thesis-Based Self-Explanation Framework

   - 3.2 Interpretability-aware code generation with post-generation
     normalization (comment/docstring stripping)
   - 3.3 Thesis generation: conditions → label

     * A = {x | cond(x) = true}, A_S = A ∩ S
     * Coverage metric: E[I(x ∈ A^x) · |A^x_S|/|S|]
     * Faithfulness (GT): E[Pr[y_true(x_i) = ŷ_0(x) | x_i ∈ A_S]]
     * Faithfulness (Code0): E[Pr[ŷ_0(x_i) = ŷ_0(x) | x_i ∈ A_S]]
     * The faithfulness gap and what it reveals
   - 3.5 Verification pipeline (condition parsing + evaluation)

4. Metrics for Interpretability
   - 4.1 Coverage: measuring explanation generality
     * Definition, computation, baseline (0.5)
   - 4.2 Faithfulness (GT): measuring data-grounded explanation quality
     * Definition, computation, baseline (1.0)
     * Why ground truth labels are the right comparison
   - 4.3 Faithfulness (Code0): measuring code-explanation alignment
     * Definition, computation, relationship to GT faithfulness
     * The faithfulness gap as a diagnostic
   - 4.4 The coverage-faithfulness tradeoff
   - 4.5 Why these metrics are sufficient and well-grounded
   - 4.6 Connection to conceptual modularity (from Tomer 2/24):
     * High coverage + high faithfulness_gt implies the code has
       modular structure where subsets of the data map cleanly to labels
     * The thesis implicitly measures how "succinct" the code's decision
       structure is without requiring any specific code format

   - 5.10 Qualitative examples (example theses, good vs. bad)

6. Discussion
   - What does "super interpretability" buy us?
   - The "same algorithm" advantage over traditional XAI
   - Dual faithfulness: what the gap tells us about learning quality
   - Limitations: cost, thesis quality variability, condition complexity
   - When does it work well vs. poorly?
   - Connection to hypothesis testing and scientific reasoning
   - Toward a theorem: if faithfulness is high, the code can be rewritten

8. RISKS & MITIGATIONS
   (Revised per Tomer meetings 2/15 and 2/24)
================================================================================

RISK 1: LLM theses might be hallucinated or unrelated to code
  MITIGATION: This is EXACTLY what the faithfulness metrics catch.
  A "random excuse" that's true about data but unrelated to the code
  will have low faithfulness_code0. A thesis that aligns with Code0
  but not reality will have low faithfulness_gt. Both metrics must be
  high for a good thesis. Even low faithfulness is a publishable finding.

RISK 2: Theses might be too specific (low coverage)
  MITIGATION: The coverage metric catches this. A thesis that says
  "if x equals exactly [these values]" will have coverage ≈ 1/|S|.
  We can tune the prompt to encourage broader conditions.
  We have a clear baseline (0.5) to compare against.

RISK 3: Classification accuracy might be worse than baselines
  MITIGATION: Phase 1 ALREADY showed accuracy is competitive (explain
  variant slightly better than standard, p=0.04). The contribution is
  interpretability metrics, not accuracy. Tomer (2/24): faithfulness
  and coverage are subsidiary to performance — they can only be good
  if Code0 performs reasonably.

RISK 4: Condition parsing might fail for complex theses
  MITIGATION: Start with simple AND/OR conditions with numeric thresholds.
  If the LLM generates unparseable conditions, log the failure and
  iterate on the prompt. Tomer: "We can actually ask the LLM to write
  its own code for verifying the conditions — it's a very short code."
  Fallback: use LLM-generated verification code with external verifier.

RISK 5: Too expensive (many API calls per experiment)
  MITIGATION: Revised experimental parameters (Phase 2.5): 1 seed,
  10 Code0 attempts for development. Thesis generation is 1/10th the
  cost of Code0 generation — invest savings in more test samples.
  Collect API keys from friends who don't use theirs (Tomer's
  suggestion). Rotate keys in a notebook.

RISK 6: Theses might just restate the code path
  MITIGATION: A code-path restatement IS a valid explanation, but it
  will have low coverage (the path is specific to one decision branch).
  The coverage metric incentivizes broader, more general theses.
  Tomer: "The thesis is a compression of what the code does for a
  specific sample." Tomer (2/24): you COULD use Code0 itself as the
  thesis — then coverage is best but faithfulness_gt equals Code0's
  accuracy. The advantage of a SHORT thesis is it can find subsets
  where faithfulness_gt = 1.0, even if coverage is lower.

RISK 7: Bias from using same LLM as learner and explainer
  MITIGATION: Tomer acknowledged this: "Is there bias there? Some LLMs
  are gonna be really good at writing explainable code but not explain
  it well, whereas the opposite can be true." This is WHY we compare
  across models — the comparison IS the finding. The verifier being
  different from the learner helps isolate this.

RISK 8: How do we verify the verifier?
  MITIGATION: Tomer: "Verifiers are common — we need to trust the system.
  We cannot solve all problems. We need to make some assumptions and try
  to solve other things." For simple boolean conditions, automated parsing
  is essentially perfect. For complex conditions, accept the limitation.

RISK 9: Faithfulness_gt might be bounded by Code0's accuracy
  MITIGATION: This is expected and desirable. If Code0 is only 70%
  accurate, faithfulness_gt can still exceed 70% IF the thesis identifies
  a "clean" subset where Code0 happens to be correct. This is precisely
  the point — the thesis finds the regions where the prediction is well-
  supported by data, even if Code0 is imperfect overall. The comparison
  between faithfulness_gt and Code0 accuracy IS interesting data.

KEY DISCOVERIES (Feb 24 - March 2, 2026):

  1. LOW REASONING IS OPTIMAL FOR gpt-5.2:
     Low outperforms medium/high on coverage (20.5% vs 16.7% vs 9.8%)
     and ties on Code0 accuracy (~59%). High uses 13x more tokens for
     equal or worse results. Mechanism: more reasoning → more complex
     code → worse generalization + more specific theses.
     Tomer's fix: "Tell it to write simple code."

  2. CODE0 ACCURACY IS THE SOLE BOTTLENECK:
     Faith_code0 ~100% across ALL datasets and settings. Faith_GT is
     limited by Code0's training accuracy. Improving Code0 accuracy is
     the single highest-leverage intervention.

  3. PIPELINE SCALES WITH PROBLEM DIFFICULTY:
     HTRU2 (easy, 76% Code0): perfect metrics.
     Mushroom (medium, 55% Code0): diverse theses, moderate metrics.
     CDC Diabetes (hard, 59% Code0): modest metrics.
     The pipeline works; the LLM's classification ability is the variable.

  4. THESIS DIVERSITY IS REAL:
     Mushroom: 20 unique multi-condition categorical theses across 25
     samples. Each describes a different region of the feature space.

  5. NARROW THESES CAN BE HIGHLY FAITHFUL:
     Individual mushroom theses: 90-100% GT faithfulness at 5-16%
     coverage. Per Tomer: "I would rather have 0.3 with a 1 than 0.5
     with a 72."

  6. SAMPLING METHOD MATTERS:
     Sequential test sampling inflated early faithfulness numbers.
     Stratified random sampling with fixed indices gives honest results.

  7. FAITHFULNESS > COVERAGE INDICATES REAL EXPLANATION:
     Faith_GT (60-68%) consistently exceeds Code0 test accuracy (55-59%)
     on covered subsets. Tomer noted this is "very positive" — the
     thesis extracts a SUBSET of the code's behavior that is MORE
     accurate than the code overall. The thesis finds the "good parts."

  8. 200 SAMPLES MAY BE TOO MANY FOR SINGLE-PASS:
     Per discussion with Tomer, LLMs struggle to write good code for
     200 samples at once. Batching (16 at a time, iteratively) may
     yield better Code0 accuracy and is more scalable.

APPENDIX: TOMER'S WHITEBOARD EQUATIONS
(2/15/2026 meeting, corrected per 2/24/2026 meeting)
================================================================================

The mathematical framework that defines the evaluation:

DEFINITIONS:
  S = training set
  ŷ_0(·) = Code0's prediction function
  y_true(·) = ground truth label from the dataset
  c_i(·) = Code1's condition checker for thesis generated for test sample x_i

  A = {x | cond(x) = true}
    The set of all samples satisfying the thesis conditions.

  A_S = A ∩ S = {x ∈ S : c_i(x) = 1}
    The subset of training data S that satisfies the conditions.

METRIC 1 — COVERAGE:
  E_{x~test}[ I(x ∈ A^x) · (|A^x_S| / |S|) ]

  Per-case:
    coverage_eq_i = 1[c_i(x_i) = 1] · |A_{x_i}| / |S|          [PRIMARY]
    coverage_ratio_i = |A_{x_i}| / |S|                            [SECONDARY]

  Aggregate:
    mean_coverage_eq = (1/n) Σ_{i=1}^{n} coverage_eq_i

  - I(x ∈ A^x) = 1 if test sample x satisfies its own thesis conditions
  - |A^x_S| / |S| = fraction of training data covered by thesis
  - Averaged over test samples
  - CRITICAL (from Tomer 2/24): must include indicator × coverage as a
    JOINT product. Do NOT condition on success and measure coverage
    separately — this is "optimistic" and fails to penalize theses that
    rarely apply. The indicator zeros out bad theses.

METRIC 2a — FAITHFULNESS, GROUND TRUTH [PRIMARY]:
  E[ Pr[y_true(x_i) = ŷ_0(x) | x_i ∈ A_S] ]

  Per-case:
    agreement_count_gt_i = Σ_{x ∈ A_{x_i}} 1[y_true(x) = ŷ_0(x_i)]

    faithfulness_gt_i =
      agreement_count_gt_i / |A_{x_i}|    if |A_{x_i}| > 0
      None                                  if |A_{x_i}| = 0

  Aggregate:
    Let I_gt = {i : faithfulness_gt_i ≠ None}
    mean_faithfulness_gt = (1/|I_gt|) Σ_{i ∈ I_gt} faithfulness_gt_i

  - For training samples in A_S, what fraction have GROUND TRUTH labels
    that match Code0's prediction on x?
  - Makes claims about the DATA: "this prediction is supported by real
    structure in the training data"
  - This is the PRIMARY faithfulness metric (from Tomer 2/24)

  WHY GROUND TRUTH (Tomer 2/24): "We want to say: forget the code.
  Look at the data. This sample is part of a group where everyone
  genuinely has the same label. We justify the prediction based on
  what we have seen in the data, not based on Code0's internal logic."

METRIC 2b — FAITHFULNESS, CODE0 [SECONDARY]:
  E[ Pr[ŷ_0(x_i) = ŷ_0(x) | x_i ∈ A_S] ]

  Per-case:
    agreement_count_code0_i = Σ_{x ∈ A_{x_i}} 1[ŷ_0(x) = ŷ_0(x_i)]

    faithfulness_code0_i =
      agreement_count_code0_i / |A_{x_i}|    if |A_{x_i}| > 0
      None                                     if |A_{x_i}| = 0

  Aggregate:
    Let I_c0 = {i : faithfulness_code0_i ≠ None}
    mean_faithfulness_code0 = (1/|I_c0|) Σ_{i ∈ I_c0} faithfulness_code0_i

  - For training samples in A_S, what fraction get the same Code0 label as x?
  - Measures alignment between thesis and Code0's actual behavior
  - Tomer (2/24): "not a terrible metric" — keep as secondary

THE FAITHFULNESS GAP:
  faithfulness_gap = faithfulness_code0 - faithfulness_gt

  - Small gap → Code0 is well-calibrated on this thesis subset
  - Large gap → Code0 has learned patterns not supported by ground truth
  - This is a novel diagnostic metric worth reporting

SUPPORTING METRIC:
  x_in_A_rate = (1/n) Σ_{i=1}^{n} 1[c_i(x_i) = 1]

  - Percentage of test samples where the thesis applies to its own sample
  - Phase 2 result: 84.4% (expected to improve with better model/thinking)

KEY INSIGHTS (Tomer):
  - Both metrics have clear baselines → we know what "good" looks like
    * Coverage baseline: ~0.5 (half the data is one class)
    * Faithfulness baseline: 1.0 (thesis claims perfect alignment)
  - These baselines make the metrics interpretable and comparable
  - The dual faithfulness (GT vs Code0) is a novel contribution:
    it reveals the gap between what the code does and what the data
    supports, which is a new way to analyze LLM-based learners
  - Coverage and faithfulness together implicitly measure how "modular"
    the code's decision structure is (Tomer's conceptual modularity
    insight from 2/24 meeting)
